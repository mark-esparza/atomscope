# SnaCleX — follow-up audit & next-steps plan (2026-06)

This audits the **current** codebase, after the seven roadmap phases and the
mmCIF / large-assembly fixes shipped. It supersedes the original external audit
(which was written from the public UI only and predates all of that work).

## Where the project stands

Compared with the original audit's "promising design, low observable production
maturity," SnaCleX is now materially hardened **on the server side**:

- 120 offline tests + CI on Python 3.11–3.13.
- Security headers + CSP, per-IP rate limiting, input sanitization, privacy/terms.
- Provenance/method blocks on every analysis; JSON/CSV/PDB exports.
- Async job queue, disk HTTP cache, grid cache.
- A redocking benchmark harness (CLI).
- Documented API (`/api/docs`), PDB **and** mmCIF support (incl. `.cif`
  fallback), and per-chain loading for mega-assemblies.

The remaining gaps are concentrated in **verification, the front end, and
operability** — exactly the areas hardest to validate in a headless sandbox.

## Findings & suggested plans (prioritized)

### P0 — Verify the UI actually works (highest)
**Gap.** `web/app.js` is ~2,400 lines and changed heavily across all seven
phases (async job polling, exports, method cards, chain picker, upload). None of
it has been exercised in a real browser — only `node --check` syntax validation.
This is the single biggest risk: a runtime regression in the render path would
be invisible to the current tests.

**Plan (Phase 7a).**
- Add **Playwright** smoke tests (the original audit named Playwright for exactly
  this) running headless in CI against a locally launched server with the
  upstream calls stubbed/recorded. Cover the golden path: load 1HSG → pick
  ligand → interactions → pockets → dock (submit + poll to result) → screen →
  each export → upload a PDB → trigger the chain picker on a large entry.
- This is a *new* dependency, but **dev-only** (CI), not shipped in the runtime —
  it doesn't violate the zero-dependency *deploy*. Gate it in a separate CI job
  so the core stdlib suite stays dependency-free.
- Interim, zero-dep step: a documented **manual QA checklist** (the flows above)
  to run before each deploy.

### P1 — Observability for a public service
**Gap (verified).** `Handler.log_message` is silenced, so there are **no access
logs and no error logs at all**. For an anonymous public service this means no
way to see traffic, errors, abuse, or latency. The original audit flagged
OpenTelemetry/Sentry; today there is nothing.

**Plan (Phase 7b).**
- Structured request logging to stdout via the stdlib `logging` module (Render
  captures stdout): method, path, status, duration_ms, IP **hash** (not raw IP —
  consistent with the privacy policy), job ids. One line per request.
- Log full tracebacks for 5xx server-side (see P2).
- Optional, env-gated Sentry/OTel exporters for teams that want them; default
  stays stdlib-only.

### P1 — Frontend XSS sweep (finish it)
**Gap.** CSP is the primary mitigation, but ~19 `innerHTML` sites interpolate
upstream-derived text (PubChem/RCSB/ChEMBL names, IUPAC strings); only a subset
are wrapped in `escapeHtml`. Defense-in-depth is incomplete.

**Plan (Phase 7c).** Audit every `innerHTML` assignment; wrap all
externally-sourced values in `escapeHtml` (or switch to `textContent`). Add a
lightweight CI grep that flags raw `${...}` interpolation of known
upstream fields. Pairs naturally with the Playwright work.

### P1 — Compute time/size budget per job
**Gap (verified: none today).** Now that a single chain can approach 99,999
atoms, the pure-Python pocket/docking routines can pin a job worker for a long
time. The async queue prevents request timeouts but not worker starvation; the
per-request budget deferred in Phase 1c is now more relevant.

**Plan (Phase 7d).** Bound the heavy routines by atom count and/or wall-clock:
e.g. cap `pockets.detect_pockets` work and docking grid size for very large
chains, and fail politely ("structure too large for pocket detection — try a
smaller chain") rather than running for minutes. Add a soft wall-clock guard in
the job runner.

### P2 — Supply chain: pin/vendor 3Dmol
**Gap (verified).** `index.html` loads `https://3Dmol.org/build/3Dmol-min.js`
**unpinned, no SRI**. A compromised or changed CDN file would execute in users'
browsers, and CSP can't catch that.

**Plan.** Best fit for the zero-dependency / offline philosophy: **vendor**
`3Dmol-min.js` into `web/` and serve it same-origin. That removes the CDN
entirely, drops the `script-src https://3Dmol.org` CSP exception, and makes the
app fully offline-capable. (Alternative: pin a versioned URL + `integrity` SRI
hash.)

### P2 — Don't leak exception text on 500
**Gap (verified).** The catch-all returns `f"Internal error: {exc}"` to the
client. Minor information disclosure.

**Plan.** Log the full traceback server-side (P1) and return a generic
"Internal error — please retry" to the client.

### P2 — `to_pdb` fidelity for edge structures
**Gap (verified).** The PDB serializer writes a single-char chain and
`{res_seq:>4}`. Multi-character mmCIF chain ids are truncated, and residue
numbers > 9,999 overflow the column and shift the line — which can mis-render
some structures in the viewer.

**Plan.** Guard `res_seq` to the field (or hybrid-36), and deterministically
remap multi-char chains to single chars (keeping a map for selection). Add a
serializer test with a >9,999 residue number and a multi-char chain.

### P3 — Chain-subset niceties
- **Conservation for subsets.** A chain subset of a real PDB knows its
  `source_pdb_id`; reuse the parent's UniProt accessions so conservation works
  for subsets (currently unavailable, like uploads).
- **Shareable subsets.** The permalink uses the synthetic `PDB-CHAIN` id, which
  isn't reloadable. Encode `pdb=<parent>&chain=<X>` in the share URL and
  reconstruct on load.

### P3 — Scientific depth (carry-overs)
- **Populate the benchmark.** Run `python -m snaclex.benchmark` against a
  PoseBusters/CrossDocked subset and commit `benchmark_results.json` so the
  methods block shows real pose-recovery numbers.
- **Screening metrics** (EF1%, BEDROC, AUROC) via the existing `summarize`
  pattern.
- **Optional Vina/GNINA track** (needs those engines + RDKit/Meeko) — still the
  highest-impact scientific upgrade, still out of scope for the zero-dep default.

### P4 — Horizontal scaling (only if needed)
In-memory jobs, caches, and the rate limiter are **per-instance**. Fine on a
single Render instance; if scaled to multiple, job ids won't resolve across
instances. The env-gated Celery/Redis track (already designed for in the
`JobManager` interface) is the answer **if and when** multi-instance is needed.

## Suggested sequencing

| Phase | Theme | Effort | New runtime deps? |
|-------|-------|--------|-------------------|
| 7a | Playwright UI smoke tests + manual QA checklist | M | dev/CI only |
| 7b | Request/error logging (observability) | S | none |
| 7c | Finish the XSS sweep + CI guard | S | none |
| 7d | Per-job compute time/size budget | S–M | none |
| 8  | Vendor 3Dmol (SRI), sanitize 500s, `to_pdb` fidelity | S–M | none |
| 9  | Subset conservation + shareable subsets | S | none |
| 10 | Populate benchmark, screening metrics, Vina/GNINA | L | opt: vina/rdkit |

**Recommended order:** 7a → 7b → 7c → 7d (verification + operability first, since
the front end is the least-verified surface), then the 8/9 polish, then 10's
scientific depth. 7a–9 are all within the zero-dependency *runtime* philosophy
(7a adds only a dev/CI dependency).
